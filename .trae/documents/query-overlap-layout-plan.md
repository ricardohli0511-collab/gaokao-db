# 查询页按钮遮挡与同类页面头图衔接治理计划

## Summary

本次计划聚焦你截图里 `/query` 页面顶部按钮被遮挡的问题，并按你的选择一并治理同类页面的头图与内容区衔接方式。

目标不是只把某个按钮“挪开”，而是把当前项目里“深色头图 + 白色内容卡片”的首屏结构从页面级负边距拼接，调整为更稳定的公共布局能力，优先完成：

1. 修复 `/query` 首屏顶部操作条被头图压住/侵入头图区域的问题。
2. 对 `PageHeader` 做局部重构，让查询页顶部操作区可以被稳定承载。
3. 同步收口首页 `/` 与院校详情页 `/school/[id]` 的同类负边距衔接模式，避免后续再次出现遮挡或错位。

本次不扩展到无关业务逻辑，不改 API、数据库、查询参数语义，也不顺带重做整个查询页功能流。

---

## Current State Analysis

### 1. 查询页真实根因

- `src/app/query/page.tsx` 在头图后直接使用 `flex-1 px-4 pb-20 -mt-6`，把整个内容区整体上拉。
- 查询页顶部操作条“筛选条件 + 普通查询/智能推荐切换”位于内容区最前面，属于被整体上拉的第一批元素。
- `src/components/PageHeader.tsx` 的 `medium` 规格只提供 `pt-12 pb-8`，底部缓冲较小。
- 因此组合结果是：查询页的顶部操作条进入头图视觉区域，形成被遮挡、被压住、可点击区域不稳定的现象。

### 2. 当前结构的系统性风险

- `src/app/page.tsx` 使用 `-mt-10`。
- `src/app/query/page.tsx` 使用 `-mt-6`。
- `src/app/school/[id]/page.tsx` 使用 `-mt-6`。
- 三个页面都在用“头图后整体上拉内容”的方式制造首屏压叠感，但没有统一的内容壳层或统一的 overlap 规则。
- `src/components/PageHeader.tsx` 本身能力较弱，只支持标题/副标题/children，没有稳定的底部承载区。
- 详情页甚至没有复用 `PageHeader`，而是在 `src/app/school/[id]/page.tsx` 里单独写了一套近似头图。

### 3. 为什么不能只补一个 `z-index`

- 这不是单纯层级问题，根因是“页面整体上拉”的布局策略本身不稳。
- 只加 `z-10/z-20` 或把 `-mt-6` 改成 `-mt-4` 只能缓解当前截图，无法保证筛选展开、切换模式、移动端换行后依然稳定。
- 查询页属于工具页，顶部操作条高度是会变化的，因此更适合“头图独立 + 顶部工具条归入稳定承载区”的结构。

## Proposed Changes

### 1. `src/components/PageHeader.tsx`

**要改什么**

- 扩展 `PageHeader` 的能力，不再只是展示标题，而是支持“头图正文 + 底部承载区”的结构。
- 新增面向查询页的底部插槽能力，例如：
  - `bottomSlot?: ReactNode`
  - 或 `footerSlot?: ReactNode`
  - 或等价的命名，但必须表达“头图底部承载的交互区”
- 为 `size="medium"` 重新定义更稳定的底部留白，不再依赖下游页面用负边距补偿。
- 调整组件内部层级，保证背景装饰层、正文层、底部承载层职责明确。

**为什么这样改**

- 查询页的问题本质上是：头图组件没为顶部工具条预留结构化位置，页面只能靠 `-mt-*` 把内容硬顶上来。
- 如果 `PageHeader` 自己能承载“按钮条/模式切换条”，查询页首屏就能从正常文档流开始，不再需要整体上拉。
- 这也为首页或后续工具页复用相同模式提供基础。

**怎么改**

- 头图仍保留当前深色渐变和装饰背景。
- 正文区保留标题/副标题。
- 在 header 下方同组件内部增加一个可选底部插槽区域。
- 插槽区域使用 `relative z-10`，并与正文区共享同一视觉上下文。
- 如果有插槽，则 `PageHeader` 自身负责额外 bottom padding / slot spacing；如果没有插槽，则保持原有简洁头图。

### 2. `src/app/query/page.tsx`

**要改什么**

- 删除当前内容区根节点上的 `-mt-6`。
- 把现在位于内容区顶部的操作条：
  - “筛选条件”按钮
  - “普通查询 / 智能推荐”切换
 迁入 `PageHeader` 新增的底部插槽区域。
- 让查询页的结果区、筛选卡、空状态、推荐区都从正常流布局开始。
- 对 query 页顶部操作条做局部重组，使其在桌面端和移动端都能稳定展示，不再被头图侵入。

**为什么这样改**

- 当前被遮挡的正是这块顶部操作条，它必须从“首屏被整体上拉的内容”变成“头图定义内的稳定结构”。
- 用户已明确希望不是只做保守挪位，而是“局部重做顶部区”，因此查询页需要在视觉和结构上同步修正。

**怎么改**

- `PageHeader` 传入：
  - `title="录取查询"`
  - `highlightChar="录"`
  - `size="medium"`
  - `bottomSlot={顶部操作条}`
- 操作条容器改成：
  - 桌面端左右分布
  - 小屏允许换行或纵向堆叠
  - 保持现有按钮语义与交互逻辑
- `filterOpen` 展开后的筛选卡不再顶进头图，而是从 header 之后自然出现。
- 普通查询空态、推荐空态、结果列表都沿用现有组件，但统一基于新的首屏起点布局。

### 3. `src/app/page.tsx`

**要改什么**

- 去掉首页内容区根节点的 `-mt-10`。
- 首页首张查询卡与头图的衔接方式改成与新的 `PageHeader` 体系一致。
- 继续保留首页“头图 + 查询卡”的视觉层次，但不再靠整个页面内容区整体上拉。

**为什么这样改**

- 你选择了“同类页一起修”，首页与查询页正在使用同一类不稳定模式。
- 如果只修查询页而首页保持 `-mt-10`，后续视觉改动仍会留下同类问题入口。

**怎么改**

- 首页不需要 query 页那样的操作条底部插槽，但需要复用新的“头图安全间距 + 内容壳层”规则。
- 保留查询卡的浮起感，改为用明确的 margin / spacing / shadow 来制造层次，而不是对整段内容使用负边距。

### 4. `src/app/school/[id]/page.tsx`

**要改什么**

- 去掉内容区根节点的 `-mt-6`。
- 重新梳理详情页头图和主体卡片之间的衔接方式，至少统一为与首页/查询页同一组 spacing 规则。
- 本次不强制把详情页整个头图完全改写成复用 `PageHeader`，但会收口它与内容区的压叠关系。

**为什么这样改**

- 详情页也在使用同一类负边距压叠模式，属于同类风险页面。
- 这次先把“布局衔接稳定性”统一掉，比顺手大改详情头图组件更符合本次范围。

**怎么改**

- 保留详情页已有的返回按钮、标签、官网链接、加入对比按钮结构。
- 只修正头图结束后主体区域的起始方式，让趋势图卡片和数据卡片在正常流里落下。
- 若需要继续保留轻微“浮起感”，只允许对首张主卡做受控上移，而不是整个页面内容区整体上移。

### 5. 视觉与交互约束

**本次保持不变**

- 深色渐变头图基调不变。
- 金色强调色不变。
- 筛选、推荐、分页、详情、对比等业务交互逻辑不变。

**本次允许优化**

- query 页顶部操作条的横向/换行策略。
- 头图与首屏之间的留白节奏。
- 首卡阴影、圆角、上下间距的一致性。

---

## Assumptions & Decisions

1. 本次核心问题是布局结构问题，不是单个按钮组件样式问题。
2. 修复范围包括 `/`、`/query`、`/school/[id]` 三类同构页面，但优先级以 `/query` 为最高。
3. 查询页顶部操作条需要局部重做结构，但不改变现有功能入口和按钮文案。
4. 不采用“继续微调负边距 + 层级”的补丁式修法。
5. 不在本次计划中扩展到：
   - API 请求逻辑重写
   - 筛选表单完全抽组件重构
   - 详情页头图彻底并入 `PageHeader`
   - CompareDrawer 底部悬浮按钮体系重做

## Verification Steps

### 查询页 `/query`

1. 进入页面默认空态时，顶部“筛选条件”按钮和模式切换完整可见、可点击，不进入头图遮挡区。
2. 展开筛选区后，顶部操作条与筛选卡之间间距稳定，白卡不压入头图。
3. 切换“普通查询 / 智能推荐”时，顶部区域不跳层、不遮挡。
4. 普通查询有结果时，结果首卡与头图之间距离自然，不发生顶住或裁切。
5. 移动端下顶部操作条允许合理换行，但不能横向溢出或互相覆盖。

### 首页 `/`

1. 首页查询卡仍保留首屏层次感。
2. 头图结束处与查询卡之间无突兀断层，也不再依赖大幅负边距。
3. 不影响首页现有按钮点击和跳转行为。

### 院校详情页 `/school/[id]`

1. 趋势图卡片与头图之间的起始位置稳定，不被头图压住。
2. 页面首屏在桌面和移动端都不出现“主体卡片顶进头图”的错位。
3. 返回按钮、官网链接、加入对比按钮的可点击性不受影响。

### 回归检查

1. `PageHeader` 在没有底部插槽的页面上仍正常渲染。
2. 头图背景装饰、圆角卡片阴影、按钮 hover 等既有视觉风格保持一致。
3. 本次修改不引入新的层级遮挡、横向滚动或空白过大问题。
